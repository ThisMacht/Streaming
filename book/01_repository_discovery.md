# Task 1: Repository Cloning and File Discovery

## Goal

Prepare a reproducible input manifest for `huggingface/accelerate` without downloading unnecessary
Git history or accidentally including non-target Python files.

## Implementation approach

`src.repo_tools.clone_repo` performs a shallow clone, reducing transfer size and setup time because
the lab analyzes the checked-out source rather than repository history. `src.repo_tools.discover_files`
walks the clone, applies path-aware exclusions, sorts paths deterministically, and writes
`data/processed/discovered_files.json`.

The current policy excludes directories named `tests`, `test`, `test_utils`, `docs`, `examples`,
`__pycache__`, and `.venv`; it also excludes `setup.py` and `test_*.py`. Comparing complete path
components avoids dropping valid names merely because they contain the substring `test`.

```bash
python -m src.repo_tools.clone_repo
python -m src.repo_tools.discover_files
```

## Evidence and result

The manifest generated by `src.repo_tools.discover_files` records the current discovered file
count and repository-relative paths. It is regenerated before the full pipeline runs, then consumed
one path at a time by the parser. The tracked pipeline output is available in
[`logs/terminal_2_pipeline_latest.log`](logs/terminal_2_pipeline_latest.log).

## Reflection

Deterministic ordering makes runs and failures easier to compare. Path-aware exclusion avoids
accidental test-file matches, and regenerating the manifest before ingestion prevents a stale list
from silently controlling the run. The exact count remains revision- and policy-dependent, so the
manifest is the source of truth rather than a number embedded in the procedure.
